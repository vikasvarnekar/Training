CUI Grid

Location

Corresponding file: Innovator/Client/Modules/cui/cuiGrid.js

Purpose

The goal of this module is to hydrate a standard Grid component with business-logic. A user can do it via so called "plugins" which will be applied to the grid as part of the options, provided to the function.

Public API

The module exports a function which can be used from outside.

function cuiGrid(control: Grid, ?options: Object): Promise(Object)

Provided options parameter may have following data within:

  • plugins - [optional] array of plugins to be applied for grid,
  • gridWrapper - [optional] instance of GridContainerWrapper to be used by plugins' logic
  • itemTypeId - [optional] id of the itemType, for which grid is instantiated
  • userPreferences - [optional] Xml holding user preferences node
  • relatedItemTypeId - [optional] id of the related itemType, for which relationshipsGrid is instantiated

Returned object contains public API of cuiGrid control:

  • destroy - handler allowing to detach all the event handlers subscribed via plugins.

How it works

CUI Grid function applies provided plugins one by one. Having this in mind, be sure, that provided plugins are in the correct order.

Existing limitations of CUI Grid:

  1. CUI Controls functions are meant to use the function the same way as it done for other controls - from CUI layout. We are not there yet
  2. You can't pluginify the grid on the go. The only way is to add desired plugin when you call cuiGrid function

How grid plugins work

Plugins are objects, which extend behavior of the grid in a particular place. Taking it simple, a grid plugin is a JS object which has access to the grid instance, options passed through cuiGrid and which can have methods which will decorate methods of grid

So this code is valid for plugins:

const myPlugin = {
    init: async function() {
        const thisPlugin = this; // "this" refers to the instance of plugin
        const targetGrid = this.grid; // grid instance can be accessed this way
        const providedOptions = this.options; // "options" provided in cuiGrid(...) are also stored in plugin

        // ...some code
    },

    setupAfterInit: function() {
        const thisPlugin = this; // "this" refers to the instance of plugin
        const targetGrid = this.grid; // grid instance can be accessed this way
        const providedOptions = this.options; // "options" provided in cuiGrid(...) are also stored in plugin

        // ...some code
    },

    getCellMetadata: function(result, headId, itemId, type) {
        const thisPlugin = this; // "this" refers to the instance of plugin
        const targetGrid = this.grid; // grid instance can be accessed this way
        const providedOptions = this.options; // "options" provided in cuiGrid(...) are also stored in plugin

        // ...some code
    }
};

How to add a plugin

As already mentioned, it is possible to pass a plugin via options during call of cuiGrid function, but in most cases you won't have an access to this code.

The most common case when you'll need a plugin is the case when there is a specific itemType which requires custom behavior to be used in context of relationships grid.

For dealing with such situations mechanism of itemType-defined plugins was introduced. This approach implies explicit relation between itemType and the plugin to be used. During instantiation of Relationships grid all associated with current itemType plugins would be automatically found and applied to the grid after default ones.

Here is how the definition of such plugin can look like:

// ------------------------------------------------------
// Innovator/Client/Modules/cui/plugins/myCustomPlugin.js
// ------------------------------------------------------

const myCustomPlugin = {
    init() {
        // some logic
    },
    getCellMetadata(cellMetadata, headId, rowId) {
        // override default behavior
    },
    // ...and so on
};

export default myCustomPlugin;

// ------------------------------------------------------
// Innovator/Client/Modules/cui/plugins/pluginDeclarations.js
// ------------------------------------------------------

import myCustomPlugin from './myCustomPlugin';
// ...other imports

const pluginDeclarations = [
    {
        itemTypeName: 'myItemType',
        plugin: myCustomPlugin
    },
    // ...other declarations
];

export default pluginDeclarations;

NOTE: In order to apply the changes made in "cui" folder, you have to recompile js-bundles. For more information, please take a look at the corresponding page.

Plugin initialization

Once you provided a grid plugin as an input parameter for a grid, it would be retreived and initialized. Initialization starts with init method of plugin, which will be invoked before any substitution is applied. This method might be asynchronous, so you can easily fetch some additional data or execute custom asynchronous check before the the run. After initializing all grid plugins setupAfterInit method will be invoked if it is defined.

init method of plugin is a good place for setting up custom plugin properties that will be used in the methods of this plugin.

setupAfterInit method of plugin can be used for setting up event subscription for custom handlers, extending grid with custom formatters/editors/sorters and configuration information of grid (see Grid page for details).

Public methods of grid overriding

Grid's methods defined in plugin will have different signature, rather than original public API methods meant for overriding, defined on the Grid class: additional first argument result is added to all the methods.

When there are multiple plugins provided, they would be applied one by one. The methods which override default behavior of the grid would be also run in a sequence, meaning that the code which provides plugins to grid is responsible for the correctness of their order.

Event handlers

There is a public API of Grid allowing to subscribe event handlers (see grid API section). But once you try to decorate grid's logic via plugins, this API could be not enough.

The main limitation of the "pure" grid's event system is that handlers are not meant to be overridden or used partially, which is very likely to be the desired result while working with plugins.

So, there is a synthetic event system build on the top of the grid and grid plugins.

The system allows you to subscribe to grid events and prevent execution of default handlers if necessary. Handlers from plugins would be subscribed in a reversed order (assumnig type, element and name of handler), so the logic from the last plugin is considered the most specific and its handlers can prevent execution of more general logic if necessary (for sequences of handlers with the same type, element and name).

See the way how you can specify event handlers using this API:

const myPlugin = {
    init() {/* ... */},
    events: [
        {
            type: 'click', // specify eventName (DOM or Grid event like "selectRow")
            element: 'cell', // specify target (the same values as for grid.on(...) allowed)
            name: 'gridLink', // specify event name...
            // ...you can break the execution of subsequent handlers named identically...
            // ...using "payload.break()" from the handler.
            method: function(payload) {
                const {name, data, break} = payload;
                const [headId, rowId, event] = data;
                // ...handler. Execution context is the instance of plugin (this === myPlugin)
            }
        }
    ],
}
