CUI ContextMenu

Location

Corresponding file: Innovator/Client/Modules/cui/cuiContextMenu.js

Purpose

CUI Context Menu module works the same way as other CUI Controls, establishing a Context Menu component according to the provided data.

Public API

Exported function has the following signature:

function cuiContextMenu(control: ContextMenu, location: String, ?options: Object): Promise(controlDestructor)

where controlDestructor returned from function is the object with:

  • show(): void method allowing to open the menu. Please note, that before the menu would be shown, visible menu entries would be calculated. If there are no visible items in the menu, the method would return.
  • destroy(): void method required for the cases, when Context Menu is no londer needed.

How it works

In order to understand better how CUI Context Menu works in scope of other controls, please take a look at CUI Layout and CUI Controls pages.

Please note, that while it is common for CUI Control to be created and initialized from CUI Layout, we are not yet there for all of CUI Context Menus. Because of this, some of CUIControls definitions describing layout structure received from server/stub don't include any information about CUI Context Menu.

There is a couple of important notes which are special about cuiContextMenu:

  1. Data received from server/stub would have a single level of nested items.

    The information about nested items is splitted in two semantical groups: data and roots. data property keeps information about nested items, while roots property only points to the ids of items to be shown on the first level below the current one.

    These properties are optional and appear only if a menu entry is a "parent menu" and has some nested entries.

    So the regular menu item would look like this:

    "id": "83C93103439243ADB2A8472FC743EB1B",
    "label": "Open",
    "on_click_handler@keyed_name": "cui_default_mwt_onViewCommand",
    "on_click_handler@type": "Method",
    "on_click_handler": "74217FBDC2554604A543BDEFA8A64ED5",
    "on_init_handler@keyed_name": "initPopupItemInItemsGrid",
    "on_init_handler@type": "Method",
    "on_init_handler": "2B56E062A54245B7851FA09CFD224882",
    "name": "com.aras.innovator.cui_default.pmig_View",

    While the menu entry with nested items would be like:

    "@typeId": "B5175846B27145EEA5653DA35ED78BE4",
    "@id": "45ACD568CFC0489BA7D5365C19208ADB",

    // specifies ids of nested items
    "roots": [
        "com.aras.innovator.cui_default.pmig_edit" // refers to the "name" of nested item
    ],
    // nested items themselves
    "data": [
        {
            "@type": "CommandBarMenu",

            "parent_menu": "45ACD568CFC0489BA7D5365C19208ADB",
            "name": "com.aras.innovator.cui_default.pmig_edit",

}

  2. All menu entries would be initialized recursively using provided on_init_handler.

  3. All image properties from the menu entry definition would be replaced with icon. This step is needed in order to keep unified interface with images loader.

  4. All the menu entries would be brought to the first level of the data structure passed to the target control. It is needed in order to keep simplicity of the items access without recursion.

  5. roots property of the item would be replaced with children (all the data preserved)

  6. data property of the item would be removed (since nested items would be stored on the first level of structure).

  7. Once all menu entries were initialized, icons required for them would be loaded automatically via SvgManager.

The last steps of initialization are passing data to the Context Menu control and returning public API. The data would be stored as a root and data properties of control.
