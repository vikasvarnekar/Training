Toolbar Component

Location

Innovator/Client/Modules/components/toolbar.js - main file with toolbar component. Public API and component definition are placed here.

Innovator/Client/Modules/components/toolbarTemplates.js - file with methods for rendering different elements within toolbar. There are templates and formatters:

  • Templates are service methods required for the rendering of toolbar itself and nested elements (abstract items).
  • Formatters are public methods required for rendering nested toolbar elements of certain type. There are default formatters for button, image, text, separator etc.

Innovator/Client/Modules/components/trimSeparators.js - utility file with a method required for proper separators rendering. The method allows to resolve cases when there are redundant separators staying nearby or at the first/last position.

Purpose

Toolbar is a Web-component which purpose is to render toolbars and toolbar-like components. If the functionality provided by default component is not enough for you, you can create your own class, using Toolbar as a base class.

Some examples of places where toolbar component is used at the moment:

  • Main header
  • Title and command bar
  • Pagination (inherited class)

Public API

Web-component API:

  • Since toolbar is an implementation of web-component, it supports all its features. Information below provides information about customizations made for toolbar, but for general info, please see MDN Web Components resource.

Constructor:

  • document.createElement('aras-toolbar'): HTMLCustomElement - create a new instance of toolbar (also <aras-toolbar></aras-toolbar>).

Inherited (from HyperHTMLElement) methods and properties:

  • .connectedCallback(): void - sets role="toolbar" (if not provided via HTML), inits overflowing behavior and initialize event listeners.

  • .disconnectedCallback(): void - removes event subscriptions.

  • .attributeChangedCallback(attrName: string): void - observes changes for role and overflow, renders updated toolbar once change happened.

  • .booleanAttributes [static, readonly] - default attributes are overflow and expanded, that means you don't have to specify their value - their presence itself defines toolbar's behavior. This property is provided by the HyperHTML library and the values provided here would be concated with the ones provided in .observedAttributes to form the complete list to be watched for changes.

  • .observedAttributes [static, readonly] - attributes which are observed for changes. By default the value is role, additionally, values from .booleanAttributes would be added to the list of observed ones. Once an attribute has been changed, .attributeChangedCallback would fire.

Methods and properties:

  • .data:

    • set Map(<key: string, value: object>) - setter for toolbar's data, accepts a Map, where a key is an identifier of the nested element, and a value is the object describing this element. After the data is set, elements would be put in containers and toolbar will automatically re-render.
    • get Map(<key: string, value: object>) - returns a Map with toolbar's elements.

  • .container:

    • set string[] - setter for a "left" container in the toolbar. Accepts an array of string keys (the same as used in data). Once data is set to toolbar, it is possible to put them either on the left or on the right within the toolbar. The items put in the container are considered "left". In most cases it is not necessary to change the value, since toolbar's elements would be placed automatically during data assignment based on the .right property of the element.
    • get string[] - getter for a "left" container.

  • .rightContainer:

    • set string[] - the same as container, but for the items on the right.
    • get string[] - the same as container, but for the items on the right.

  • .on(eventType: string, callback: Function): Function - method allowing to subscribe to events. Each time an event happens, the callback would fire with the information about the element on which the event had happen. If there is no element (e.g. you've clicked on the toolbar's padding area or between buttons) or the element is disabled, the callback wouldn't fire. The signature of the callback: (itemId: string, event: Event) => void | Promise<void> The method returns a detach function, allowing to remove created subscription.

  • .render(void): Promise - method for toolbar rendering. The method renders the toolbar according its current state and data. In most cases you don't have to call this method explicitly, since all data changes cause automatic re-rendering. The method works asynchronously, so if you rely on the toolbar appearance in DOM, consider awaiting of the returned Promise.

  • static .extendFormatters(Object) - allows to provide additional formatters for toolbar. Accepts an object. Properties' names of the object would be used as identifiers of element's type. Corresponding function would be used as formatter. Formatters have the following signature: (item, options) => VirtualNode.

Properties providing access to the inner structures of the toolbar:

  • .datastore - the property providing direct access to the data map. This property is useful for usage in derived classes, because it doesn't imply automatic rendering. NOTE: This property is not recommended for usage from the outer code.

  • .templates - the property providing direct access to the defaultTemplates, which are responsible for rendering of the toolbar itself and starting rendering of its elements. This property is useful for usage in derived classes. NOTE: This property is not recommended for usage from the outer code.

  • .focusedItemId - the property providing access to the key of currently focused item within toolbar. This property is useful for usage in derived classes to work with a11y-related functionality. NOTE: This property is not recommended for usage from the outer code.
