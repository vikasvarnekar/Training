Project Structure

Structure of the client-related files

There are several folders which are used by client teams during development. If some folders present in the repo, but not listed here it means that they are either new or rarely used:

Innovator.git/
    AML-packages/ -- data structures and logic (methods) which would be stored in DB are placed here as .xml files
    Innovator/
        Client/
            buildScripts/ -- scripts for environment setup for FE development
            flow-typed/ -- flow type definitions
            images/ -- icons and images storage
            javascript/ -- legacy js-codebase. A lot of public API modules are in this folder
                IncludeNamespaceConfig.xml -- map defining how js files are concatenated together for different areas
            jsBundles/ -- the folder where all bundled js-files are stored. No manual work here, each file is built
            Modules/ -- domain-organized codebase
                aras/ -- business-logic related to the main innovator window (not any specific area)
                aras.innovator*/ -- business-logic related to a specific domain area
                components/ -- reusable, pure components, not bound to any context
                core/ -- core logic which couldn't be treated as area-specific, services
                cui/ -- logic related to handling CUI-structures and associating this data with layouts, controls, logic-processors
                NavigationPanel/ -- business-logic related to the main navigation panel
                polyfills/ -- customized polyfills
                store/ -- files related to Single Item Store, reducers, actions etc.
                jsBundleList.json -- mapping of modules to the compiled jsBundles
            nodejs/ -- tools for bundling and processing client code, compilation procedure is defined here
            scripts/ -- mostly *.html pages and corresponding resources for different views. Domain-organized
            Solutions/ -- custom projects made as Innovator extensions
            styles/ -- styles and xslt-templates are defined here. Control/Domain-organized
                less/ -- styles for components and some domain-areas, BEM-notation is preferred here
            tests/ -- folder with unit-tests related files
                AML-packages/ -- methods from "Innovator.git/AML-packages" are extracted from xml-files, wrapped as functions and placed here. File names match original ones. "npm run build-aml-methods" for populating
                coverage/ -- generated code-coverage is stored here
                css/ -- html-pages for some controls are stored here for checking their styles, not automated
                junit/ -- raw coverage data
                storybook/ -- dynamic web-page with controls for checking their styling and interactivity
                testCases/ -- unit-test files. Could be named as: 1) {file}.js - legacy naming; 2) {file}.spec.js - non-compilable unit test; 3) {file}.spec.es.js - compilable (modules+modern syntas) and bundlable; 4) {file}.spec.amd.js - non-compilable AMD-modules-based tests
                testPages/ -- dynamic web-pages with controls for checking their styling and interactivity (used as sandboxes before storybook)
                htmlUnitTests.js -- entry point for "npm run test-html"
                javascriptTests.js -- entry point for "npm run test" & "npm run test-dev", Karma-based
            vendors/ -- copied 3rd party libraries (node_modules are not distributed)
            jsCompileAll.js -- entry point for "npm run build"
            jsWatcher.js -- entry point for js-files of "npm run watch"
            package.json -- downloaded modules and scripts
    Tests/
        ComponentTests/ -- framework for running of component tests is stored here alongside test-scenarios and stubbed data
            baseline/ -- Stubbed responses organized by test-scenario name are stored here: 1) *.baseline files - matched hashed_request-hashed_response data objects; 2) *.order files - meant to define the correct order of responses for the same request which happens multiple times
                CUI/ -- stubbed data-responses related to CUI
                InnovatoServer/ stubbed data-response for requests related to InnovatorServer.aspx
            junit/ -- produced output folder with execution report
            preTestAMLs/ -- contains AML-files that contain AMLs for preliminary preparation of the state of the Innovator database required for execution of particular test-scenario
            src/
                /baseline -- files related to the logic of the stubbing of server responses (baseline-generation)
                /common -- modules related to common operations
                /compoonents -- modules that encapsulate logic of interaction with different controls
            test/ -- test-scenarios
            testOptions.js -- configuration file for test runner
