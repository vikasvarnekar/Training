Development Principles

The most important principles of Innovator Client development

  • Compatibility. While working on a long-term product, our goal is to support customers as much as possible. It means that we have zero-tolerance to changes that break something that was in use before. It doesn't mean that we are not going to rework our codebase, it means that we should care about the way our code could be used by others and provide a clear tools and guidance for migration to the newest implementation. Important: we never change our public API or any DB-related code in major updates (service packs).

  • Supportability. The code we write is created not for machines, but for people who will be working here after us. Clear and understandable solutions should be preferred over sophisticated and highly-abstract ones.

    • New code must be covered with unit-tests. Tests should be as close to the real business-cases as possible, yet staying unit ones.
    • If you can provide the documentation alongside your code - please do it.
    • If you know that a certain code snippet is a tricky one because of any reason - please provide information about it in a commit message related to the change or via comment.

  • Reusability. If there is a real chance that the same functionality would be needed somewhere else, you should consider creating reusable code. The main technique is a separation between general logic and business-logic:

    • General logic. The logic which can be used somewhere else in the exact same way or with context-dependent customizations. A good example of a general UI-component is a toolbar - it could be used anywhere in the project and could be placed there for whatever aims: it can be header, seacrch bar, pagination or loading progress indicator. As an example of a general logic-related module, let's say it could be a module for dealing with network requests or file loader. So, when creating such sort of controls we should:

      • keep it as simple as possible - consider YAGNI principle while adding new functionality
      • keep it as pure as possible - it should be possible to run the component in a pure HTML page without any context. Why? Because it is meant to be used anywhere.
      • keep it as determined as possible - result of the above concepts - Single Responsibility if you want.

    • Business-logic. The logic which is needed in the exact place. It knows about its surroundings and communicate with them. It is almost everything that should be done on click here and on keypress there. It is about interaction between different parts of the product. This logic doesn't make any sense when we extract it from the context.
